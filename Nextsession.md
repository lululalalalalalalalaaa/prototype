## 1. Stage 2(BM25/하이브리드) 재검토  ★ (이번 측정이 드러낸 것)

- **왜**: 확장 골든셋에서 **하이브리드 이득이 미미**(dense 0.821 → hybrid 0.825), easy·hard는 오히려 소폭 하락.
  현실 질의는 정확 토큰이 아니라 구어·동의어라 BM25가 약하고, 리랭커가 대부분 흡수한다. 옛 46세트가 BM25를
  과대평가했다.
- **무엇**: ① BM25를 **드롭**(dense→rerank만)했을 때 vs 유지의 ablation 측정, ② 또는 BM25 가중/토크나이저 개선
  (형태소·동의어 사전). 단순화가 이기면 파이프라인에서 BM25 제거.
- **성공 기준**: rerank MRR 0.962 **비회귀**하면서 BM25 제거로 단순화(또는 개선으로 hybrid가 유의하게 이김).

## 2. hard 질의 개선 + near-domain 그라운딩 강화  (남은 헤드룸)

- **왜**: ① hard MRR 0.925 — 지역 광역명(영남/호남/경기)·물 종류 경계에서 일부 실패. ② **near-domain 기권
  0.867** — 지하철·수소차·화물자동차·농업용수처럼 *데이터에 가깝지만 없는* 질의에 2건 오추천(46세트 1.000이
  더 어려운 세트에서 드러난 약점). Stage 6의 "기권 낮으면 강화" 조건이 이제 (약하게) 충족됨.
- **무엇**: rerank/recommend 프롬프트에 도메인 힌트(지역 동의어·용수 종류 정의·"비슷해 보여도 다른 종류면 no_match").
  메타데이터 활용 가능. **measure→강화→재측정**(`--mode answer`/`rerank`)으로 검증.
- **성공 기준**: hard MRR > 0.925 · near-domain 기권 > 0.867, **응답 정확도 0.974·과잉기권 0 비회귀**.

## 3. 커밋 & 배포 정리  ★

- **왜**: 이번 세션 전부 미커밋. 이력 남기고 배포 절차 확정 필요. **index/는 private repo 커밋으로 결정됨**.
- **무엇**: Conventional Commits로 묶어 커밋. private 저장소에 `index/` 포함 → Streamlit Cloud(또는 자체 서버) 연결.

## 4. rerank + recommend 호출 병합  (지연·비용)

- **왜**: 검색 1회당 OpenAI 호출 3회(임베딩→리랭커→추천). `--mode answer`가 느린 원인.
  리랭커와 추천을 1회 LLM 호출로 합치면 검색당 2회로 준다.
- **무엇**: top-N을 한 번에 "재정렬 + 추천 + no_match"로 처리하는 단일 프롬프트(또는 리랭커가 추천까지 산출).
  본문 컨텍스트 다이어트도 함께.
- **성공 기준**: rerank MRR 0.962 · 그라운딩 1.000/0.974 **비회귀**, 검색 지연·비용 감소(실측).

## 5. 검색 속도 — numpy 벡터화  (규모 확장 시)

- **왜**: 순수 파이썬 코사인은 수백 문서×청크(수천~수만 벡터)에서 느려진다(현재 37문서라 무문제).
- **무엇**: `retrieve.cosine_similarity`/`hybrid_rank`를 numpy 행렬연산으로 교체(임베딩은 이미 npz).
  eval-가드(동작 보존) 하에 진행.
- **성공 기준**: 목표 규모에서 질의 지연 < 목표치, eval 지표 **불변**.

## 6. 관측성 / 로깅  (원 Stage 7)

- **왜**: 운영 중 질의 분포·점수·지연·비용·실패를 추적해야 임계값 드리프트·회귀를 잡는다.
- **무엇**: 질의·회수 청크·점수·추천·지연·토큰을 구조화 로깅(파일/sqlite). 주기적 eval 재측정 루틴.
- **성공 기준**: 로그로 "최근 N건 검색의 품질·비용"을 집계 가능.

## 7. 메타데이터 활용 — 필터 UI  (UX)

- **왜**: Stage 5에서 세부정보(기능단위·시스템경계·Climate change_Total 등)를 이미 인덱스에 precompute함.
- **무엇**: 지역/기능단위 등으로 결과 필터, 추천 카드에 핵심 수치(Climate change_Total) 노출.
- **성공 기준**: 필터·수치 표시가 정확히 동작(메타데이터 기반).

## 8. (옵션) floor·no_match 로직 정리

- **왜**: 그라운딩 측정 결과 LLM no_match가 완벽(1.000)하므로, 코사인 `similarity_floor`(0.40)는 이제
  '다른 유사 후보' 표시용으로만 남았다. 의미를 문서화하거나 단순화 여지.
- **성공 기준**: floor 역할이 명확히 문서화되거나, eval 비회귀로 단순화.
