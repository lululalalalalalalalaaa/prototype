## 1. hard 질의 개선  (남은 헤드룸 — 최우선)

- **왜**: hard MRR 0.925 — 지역 광역명(영남/호남/경기→수도권)·물 종류 경계에서 일부 실패. 여기가 다음 품질 표적.
- **무엇**: rerank 프롬프트에 도메인 힌트(지역 동의어·용수 종류 정의), 메타데이터(지역·기능단위) 활용. 난이도별
  분해로 정확한 실패를 짚어 개선. (참고: BM25는 net-neutral이라 여기엔 도움 안 됨)
- **성공 기준**: hard MRR > 0.925, 전체·그라운딩 비회귀.
- ✅ **완료된 관련 작업**: BM25 재검토(ablation: dense풀/hybrid풀 모두 0.962, net-neutral → **유지**),
  near-domain 그라운딩 강화(recommend 프롬프트 → 기권 0.867→**1.000**, 과잉기권 0.013).

## 3. rerank + recommend 호출 병합  (지연·비용)

- **왜**: 검색 1회당 OpenAI 호출 3회(임베딩→리랭커→추천). `--mode answer`가 느린 원인.
  리랭커와 추천을 1회 LLM 호출로 합치면 검색당 2회로 준다.
- **무엇**: top-N을 한 번에 "재정렬 + 추천 + no_match"로 처리하는 단일 프롬프트(또는 리랭커가 추천까지 산출).
  본문 컨텍스트 다이어트도 함께.
- **성공 기준**: rerank MRR 0.962 · 그라운딩 1.000/0.974 **비회귀**, 검색 지연·비용 감소(실측).

## 4. 검색 속도 — numpy 벡터화  (규모 확장 시)

- **왜**: 순수 파이썬 코사인은 수백 문서×청크(수천~수만 벡터)에서 느려진다(현재 37문서라 무문제).
- **무엇**: `retrieve.cosine_similarity`/`hybrid_rank`를 numpy 행렬연산으로 교체(임베딩은 이미 npz).
  eval-가드(동작 보존) 하에 진행.
- **성공 기준**: 목표 규모에서 질의 지연 < 목표치, eval 지표 **불변**.

## 5. 관측성 / 로깅 — 영속화  (원 Stage 7, 일부 완료)

- **왜**: ✅ 검색당 단계별 trace(타이밍·토큰·USD)·출처·콘솔 로깅·인앱 패널은 **완료**. 남은 건 **영속 로그**
  (질의·점수·비용을 파일/sqlite에 누적)와 주기적 eval 재측정 루틴.
- **무엇**: `pipeline.search`의 trace를 파일/sqlite에 append. "최근 N건 품질·비용" 집계 뷰.
- **성공 기준**: 운영 중 누적 로그로 비용·품질 드리프트 추적 가능.

## 6. 메타데이터 활용 — 필터 UI  (UX)

- **왜**: Stage 5에서 세부정보(기능단위·시스템경계·Climate change_Total 등)를 이미 인덱스에 precompute함.
- **무엇**: 지역/기능단위 등으로 결과 필터, 추천 카드에 핵심 수치(Climate change_Total) 노출.
- **성공 기준**: 필터·수치 표시가 정확히 동작(메타데이터 기반).

## 7. (옵션) floor·no_match 로직 정리

- **왜**: 그라운딩 측정 결과 LLM no_match가 완벽(1.000)하므로, 코사인 `similarity_floor`(0.40)는 이제
  '다른 유사 후보' 표시용으로만 남았다. 의미를 문서화하거나 단순화 여지.
- **성공 기준**: floor 역할이 명확히 문서화되거나, eval 비회귀로 단순화.
